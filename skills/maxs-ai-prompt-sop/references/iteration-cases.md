# 麦壳思提示词迭代案例库

每个案例都是"踩坑—过度纠正—简化收敛"的完整记录。这些是教学的真正核心,比成功案例更值钱。

---

## 案例 1:雪山古建白底资产陈列图(三步迭代经典)

### 目标
出一张雪山主题的国风建筑资产陈列图,白底无阴影,用作教学素材

### 第一版:试水(暴露问题)

**提示词**:
```
把图中的雪山古建场景的建筑结构部件按照3D资产图一次摆放,
8个建筑大件不能重复,按照大中小依次排放,
重点表现飞檐、斗拱、廊柱、石基、瓦片,
8k分辨率,顶级摄影照片,俯视35度视角,
电影级布光,黄昏暖色侧光搭配冷色环境光,
风格化PBR材质,辅以手绘质感的木材风化和冰雪覆盖细节,
黑神话悟空风格,asset showcase陈列图
```

**结果**: 出图是深色背景 + 黄昏氛围,**不是白底**

**问题分析**: 提示词里没提背景,模型按"黄昏光"自动脑补了对应的深色环境氛围

### 第二版:过度纠正(更糟)

**提示词**(在第一版基础上加约束):
```
... [前面省略] ...
纯白色无缝背景,RGB(255,255,255),
背景如Apple官网产品图、电商主图、博物馆图录般纯净,
物件悬浮陈列,无地面、无投影、无接触阴影、无环境反射、
无背景渐变、无背景纹理、无氛围光晕,
均匀柔和的棚拍布光,全方位无影灯,
中性白光5500K,无暖色染色,无冷色染色,
保留物件本身的木材原色、瓦片青灰、积雪冷白、冰挂透明感,
... [后面省略] ...
```

**结果**: 白底成功了,但**物件被漂白,失去原色**,而且**出现"幽灵半成品"bug**——中间漂浮一个没画完的飞檐残片

**问题分析**: 
- `无影灯` + `中性白光5500K` 把模型逼到"产品摄影"模式,过度规训
- 太多"无 XX"的负面指令累积冲突
- 数量给到 20 + 约束过多,触发 bleeding bug

### 第三版:简化收敛(成功)

**提示词**:
```
把图中的雪山古建场景结构部件按照3D资产图一次摆放,
20个结构部件不能重复,按照大中小依次排放,
8k分辨率,顶级摄影照片,俯视35度视角,
电影级布光,黄昏暖色侧光搭配冷色环境光,
风格化PBR材质,半写实搭配法线贴图层次,
辅以手绘质感的磨损、积雪、冰挂、风化细节,
保持物件比例关系准确,
黑神话悟空风格,asset showcase陈列图,白背景。无阴影
```

**结果**: 完美。白底干净、物件原色保留、25+ 件资产完整、风格统一

### 反直觉发现(教学黄金点)

1. **黄昏光不会污染白底** —— 之前误以为"光照指令会决定背景颜色",其实模型对"白背景"这种独立指令的响应优先级高于"光照氛围联想"
2. **`白背景。无阴影` 五个字 > 一大段精确描述** —— 简洁的指令在权重上反而比堆砌精确描述更高
3. **位置很重要** —— 同样的"白背景"指令,放在中间和放在最后,效果不一样。放最后是"收尾指令",权重更高
4. **加约束不一定改善,可能恶化** —— 过度规训会让模型出图僵硬、概率性 bug 增加

### 教学使用建议

这个案例作为"AI 提示词工程入门"的开篇案例,**比任何一次性成功的提示词都有教学价值**。让学生看到"我以为加约束会更好,结果反而更差"的反直觉过程,这是市面 90% AI 教程不会教的。

## 案例 2:从场景图到资产拆解的完整工作流

### 目标
给一张完整场景概念图(月台、雪山古建、街区等),拆解成可用于 3D 重建的独立资产清单

### 错误工作流(学生常犯)

直接说 "把这张图的所有东西拆出来" —— 模型不知道你要 8 件还是 80 件,也不知道你要白底还是带氛围,出图随机性极大

### 正确工作流(三步)

**第一步: 让 AI 先列清单**

```
列出这张场景图中所有可建模的独立 3D 物件,
按以下分类输出: hero props / mid-ground props / set dressing / 
architecture / foliage / small details。
输出为编号清单
```

**第二步: 按数量分组拆解**

不要一次拆完。**分 2-3 张图拆**,每张 15 件以内,质量明显比 30 件一张高:
- 第一张: 建筑大件(套黄金模板)
- 第二张: 中件 + 装饰(套黄金模板)  
- 第三张: 自然元素 + 小道具(套黄金模板)

**第三步: 单件精修**

挑出最满意的几件,用模板 2(单件 3 视图)单独再出一遍,喂 Tripo/Hunyuan3D 重建

### 验证数据

- 一次性拆 25 件: 平均 1.8 个 bug / 张,可用率 60%
- 分批拆 3×12 件: 平均 0.3 个 bug / 张,可用率 90%
- 但消耗的积分: 分批 > 一次性(约 2.5 倍)

### 教学建议

让学生明白**质量和成本的权衡**: 
- 个人探索用一次性,省积分
- 教学素材 / 客户交付用分批,保质量
- 不要让学生陷入"我要一张完美的 30 件拆解图"的执念

## 案例 3:风格漂移问题(中式飞檐变日式神社)

### 现象

提示词写"中式古建"、"飞檐",但 Banana Pro 经常出来日式神社风格的屋顶(平直、檐口不上翘、装饰简化)

### 原因

Banana Pro 训练数据中日式建筑权重可能高于中式,"飞檐 / curved roof / Asian architecture" 这些关键词在它的特征空间里更靠近日式

### 应对策略

**A 级方案(精确锚定)**:
```
中国传统斗拱结构,飞檐高翘上扬,非日式神社风格,
参考故宫太和殿、应县木塔的飞檐曲线,
歇山顶或庑殿顶,而非日式入母屋造
```

**S 级方案(配图垫图)**:
- 提示词改成图生图模式
- 喂一张参考图(黑神话悟空截图 / 故宫照片 / 燕云十六声场景)
- 提示词主导风格,垫图主导结构

### 通用规律

凡是"中式 vs 日式"、"北欧 vs 美式"、"地中海 vs 西班牙"这种**容易混淆的近邻风格**,都要用"是 X 不是 Y"的对比锚定法,不要只说"是 X"

## 案例 4:数量与质量的甜蜜点

### 测试数据(Banana Pro,资产陈列图场景)

| 请求数量 | 实际出现数量 | bug 率 | 可用率 |
|---------|------------|-------|-------|
| 5-8 件 | 5-10 | 极低 | 95% |
| 10-12 件 | 12-15 | 低 | 90% |
| 15-20 件 | 18-25 | 中等 | 80% |
| 25-30 件 | 22-28 | 高 | 60% |
| 35+ 件 | 25-30(达不到) | 极高 | 40% |

### 关键发现

- Banana Pro 实际会**比要求的数量多画 20-30%** —— 这是它的特性
- 请求 20 件,实际出 25 件左右,这不是 bug
- 但请求 35 件以上,它反而画不到 30 件,触发"摆烂"模式
- **甜蜜点是请求 15-20 件**,实际出图 18-25 件,质量与数量平衡最好

### 教学建议

不要让学生看到大佬出 30 件就也写 30 件,效果会很差。教他们**根据用途选数量**:
- 教学演示: 8-12 件,每件都看得清细节
- 项目美宣: 15-20 件,信息密度饱满
- 资产库参考: 多张 15 件,不是一张 50 件

## 新增案例规范

新案例进入这个库,必须包含:

1. ✅ 完整的三阶段记录(试水/纠错/简化),不是只写成功版
2. ✅ 失败版必须保留,这是教学价值所在
3. ✅ 反直觉发现(至少 1 条)
4. ✅ 教学使用建议(怎么用在课程里)
5. ✅ 实际测试数据(如果有)

只有成功结果、没有失败过程的"案例"不算案例,只是炫技。
